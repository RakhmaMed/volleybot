# Руководство по тестированию VolleyBot

Это руководство описывает актуальный способ тестировать бота локально без риска для основной БД и продового чата.

## Быстрый старт

Самый простой путь:

```bash
python test_bot.py
```

`test_bot.py`:
- создаёт временную SQLite БД
- выставляет `VOLLEYBOT_DB_PATH` только на время процесса
- заполняет БД тестовыми залами, игроками, подписками и кассой
- запускает бота в polling-режиме
- удаляет временную БД после остановки

## Что нужно заранее

Нужен рабочий `.env` с минимумом:

```env
TELEGRAM_TOKEN=...
CHAT_ID=...
ADMIN_USER_ID=...
```

`test_bot.py` использует тот же токен и те же базовые настройки, что и основной бот, но работает на отдельной временной БД.

## Какие тестовые данные создаются

При запуске `test_bot.py` в БД автоматически создаются:

### Залы

| Зал | Разовая игра | Месяц | Подписчики |
|---|---:|---:|---|
| Понедельник | 150₽ | 6000₽ | Алиса, Карина |
| Среда | 0₽ | 0₽ | — |
| Пятница | 150₽ | 4500₽ | Борис, Карина |

### Игроки

| Игрок | Баланс |
|---|---:|
| Алиса | -300₽ |
| Борис | -150₽ |
| Карина | 0₽ |

### Касса

| Показатель | Значение |
|---|---:|
| Касса | 2500₽ |

Примечание:
- внутри БД связи залов теперь идут по `poll_template_id`
- в Telegram-интерфейсе по-прежнему показываются названия залов

## Рекомендуемый smoke-test

После запуска `python test_bot.py` откройте бота в Telegram и последовательно выполните:

```text
/start
/help
/schedule
/balance
/subs
/player
```

Ожидания:
- `/schedule` показывает три тестовых зала
- `/balance` показывает кассу и балансы игроков для администратора
- `/subs` показывает предзаполненные подписки по дням
- `/player` показывает игроков и подробные балансы

## Финансовые сценарии

### Сценарий 1. Оплата игрока

```text
/balance
/pay Алиса 300
/balance
```

Ожидание:
- баланс Алисы: `-300₽ -> 0₽`
- касса: `2500₽ -> 2800₽`

### Сценарий 2. Восстановление без кассы

```text
/balance
/restore Борис 150
/balance
```

Ожидание:
- баланс Бориса: `-150₽ -> 0₽`
- касса не меняется

### Сценарий 3. Оплата зала

```text
/pay Оплата зала
```

Ожидание:
- бот показывает inline-кнопки для неоплаченных платных залов за текущий месяц
- доступны `Понедельник` и `Пятница`
- после нажатия кнопки бот списывает сумму из кассы
- повторная попытка оплатить тот же зал за тот же месяц блокируется

Проверка:

```text
/balance
/pay Оплата зала
```

После оплаты одного зала в списке должен остаться только второй.

## Месячный опрос

Для ручного тестирования абонемента используйте:

```text
/open_monthly
```

Дальше:
- проголосуйте за нужные залы
- если хотите проверить деление стоимости, проголосуйте с нескольких аккаунтов

Завершение:

```text
/close_monthly
/balance
/subs
```

Ожидание:
- опрос закрывается
- бот считает суммы по подписчикам
- с проголосовавших списываются деньги
- касса не меняется

Важно:
- `/open_monthly` и `/close_monthly` доступны только администратору
- лучше не держать параллельно открытый обычный опрос, пока тестируете месячный

## Личка и группа

### Личные сообщения

Можно проверить:
- `/help`
- `/schedule`
- `/balance`
- `/start`

Часть сценариев с inline-кнопками и групповыми опросами удобнее проверять в группе.

### Тестовая группа

Рекомендуемый вариант для полного ручного теста:
1. Создайте отдельную группу.
2. Добавьте туда бота.
3. Дайте боту права администратора.
4. Убедитесь, что ваш аккаунт считается админом группы.
5. Запустите `python test_bot.py`.

Это позволит проверить:
- inline-кнопки
- платежи из кассы
- месячный опрос
- взаимодействие нескольких пользователей

## Получение Chat ID

Если нужен отдельный тестовый чат:

```bash
python get_chat_id.py
```

Дальше:
1. Отправьте боту сообщение в личке или группе.
2. Возьмите `CHAT_ID` и `ADMIN_USER_ID`.
3. Обновите `.env`, если хотите тестировать на другом чате.

## Автоматические тесты

### Полный набор

```bash
uv run pytest -q
```

### Только БД и финансы

```bash
uv run pytest tests/test_db_polls.py tests/test_fund_and_subscriptions.py -q
```

### Только обработчики

```bash
uv run pytest tests/test_handlers.py tests/test_balance_handlers.py -q
```

### С покрытием

```bash
uv run pytest tests/ --cov=src --cov-report=html
```

Отчёт будет в `htmlcov/index.html`.

## Отладка

### DEBUG-логи

В `test_bot.py` уже включён `DEBUG`-уровень логирования. Этого обычно достаточно.

Если запускаете обычный бот, можно явно задать:

```bash
LOG_LEVEL=DEBUG python -m src.bot
```

### Если не работают inline-кнопки

Проверьте:
1. Бот действительно запущен.
2. Команда `/pay Оплата зала` возвращает сообщение с кнопками.
3. После нажатия в логах появляется `callback_query`.
4. Команда `/webhookinfo` не показывает проблем с pending updates / last error.

### Если бот не отвечает в группе

Проверьте:
1. Бот добавлен в группу.
2. У бота есть админские права.
3. Ваш аккаунт является администратором.
4. Используется правильный `CHAT_ID`.

### Если хотите сохранить тестовые данные

`test_bot.py` удаляет временную БД после остановки. Если нужно сохранить состояние между запусками, запускайте обычный бот с отдельной БД:

```bash
VOLLEYBOT_DB_PATH=data/test.db python -m src.bot
```

Перед этим убедитесь, что:
- вы используете тестовый чат
- вы не указываете путь к продовой БД

## Лучшие практики

### Делайте

- используйте `test_bot.py` для быстрого ручного тестирования
- тестируйте финансовые сценарии в отдельной группе
- прогоняйте `uv run pytest -q` перед важными изменениями
- проверяйте `/balance` до и после `/pay`, `/restore`, `/pay Оплата зала`
- для месячного опроса проверяйте и голосование, и финальные списания

### Не делайте

- не тестируйте новые сценарии на продовой БД
- не запускайте ручные эксперименты на основном боте без необходимости
- не опирайтесь на старые команды вроде `/open 1` или `/close 1` — их больше нет
